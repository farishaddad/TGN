Layer 1 — Transaction Ingestion
Amazon Kinesis Data Streams
Event ingestion · On-Demand mode
Why needed: Real-time event ingestion at 1K–10K TPS with ordering guarantees within a shard. On-Demand mode auto-scales without pre-provisioning shards. Kinesis provides replay capability — if a model update requires reprocessing historical events, you can replay from the stream (up to 7-day retention). This is also the trigger for graph construction: each transaction event becomes a Neptune edge via the downstream Lambda.
Ingestion
AWS Lambda — Graph Event Processor
Kinesis consumer · graph construction
Why needed: Consumes from Kinesis, transforms each raw transaction into a graph event (node lookups, edge feature encoding via PRAGMATimeEncoder), and writes to Neptune. Stateless and serverless — cost-proportional to volume, scales to zero between bursts. Each Lambda invocation processes a Kinesis shard batch; parallelism is controlled by the shard count.
IngestionGraph
Layer 2 — Graph Store
Amazon Neptune (database)
Persistent property graph · Gremlin API
Why needed: Authoritative store for the full transaction graph — cardholders, merchants, devices, account-change events as property graph nodes and timestamped directed edges. Neptune is chosen over a relational DB because it natively handles multi-hop Gremlin traversals needed by TGN-SEAL (k=2 hop subgraph extraction) and ETGAT (fund-flow chain queries). At inference, only the ~3% of flagged transactions trigger Neptune subgraph extraction — the fast path uses Redis instead.
GraphInference (Pass 2)
Amazon Neptune Analytics
In-memory analytics · vector search
Why needed: Separate from Neptune DB — optimised for analytical queries on the full graph. Powers community detection (surfaces fraud rings automatically), vector similarity search on TGN embeddings ("find transactions similar to this flagged one"), and interactive GraphStorm notebook workflows for analysts. Neptune Analytics runs the graph analytics layer; Neptune DB handles the OLTP write path. Used offline (not on the scoring hot path).
GraphAnalyst Tooling
Layer 3 — Node Memory Store (TGN per-entity state)
Amazon ElastiCache for Redis (Valkey)
TGN memory + batch embeddings · <2ms lookup
Why needed: The BRIGHT Lambda architecture (eBay, CIKM 2022) separates expensive multi-hop computation (batch, hourly) from real-time scoring (sub-2ms lookup). Redis stores two data types: TGN memory vectors s_i(t) (128-dim, updated after each transaction, TTL=90 days inactivity) and hourly batch embeddings from full neighbourhood aggregation (TTL=2h). Neptune P99 Gremlin latency is 50–200ms — Redis is what keeps the fast path under 20ms for 50M+ active entities.
MemoryInference
Amazon SageMaker Feature Store
Profile features + velocity + drift stats · <10ms online
Why needed: Stores three categories: (1) Node profile features for ProfileStateEncoder — account age, balance quantile, spending limit (inputs to the PRAGMA-inspired static branch); (2) rolling velocity counts at 5min/1hr/24hr; (3) deviation reference stats — rolling mean/std of anomaly scores per card for the TFLAG two-hurdle filter. Keeping these in Feature Store (vs. Redis) provides offline/online parity for training — a common source of AUC degradation without it.
MemoryTraining
Layer 4 — ML Inference (Two-Pass)
SageMaker Real-Time — TGN Fast Path
ml.g4dn.2xlarge × 3 · Pass 1 · 100% of transactions
Why needed: Hosts the Pass 1 pipeline: retrieve embeddings from Redis → run PRAGMATimeEncoder delta → GraphAttentionEmbedding → RFScoringHead. GPU is needed for the attention computation even at inference. Three instances provide HA across 3 AZs. Pass 1 approves ~97% of transactions at ~20ms without any Neptune graph traversal. Score <0.3 → approve; score ≥0.3 → route to Pass 2.
Inference Pass 1
SageMaker Real-Time — TGN-SEAL Structural
ml.g4dn.xlarge × 2 · Pass 2 · ~3% of transactions
Why needed: For flagged transactions, extracts the k=2 enclosing subgraph from Neptune, assigns DRNL structural labels (dist_to_card, dist_to_merchant per node), concatenates with TGN embeddings, and classifies via DGCNN. Detects bridge edges, triangle closure with fraud clusters, and novel card–merchant structural positions. Separate endpoint prevents SEAL from consuming fast-path GPU capacity — it only sees 3% of volume, so smaller instances suffice.
Inference Pass 2Graph
SageMaker Batch Transform — Batch Embedder
ml.g4dn.12xlarge · hourly · BRIGHT Lambda batch layer
Why needed: The "batch layer" of the BRIGHT Lambda architecture. Runs hourly: takes all transactions from the last hour, runs full multi-hop TGN neighbourhood aggregation (expensive — traverses up to k=2 hops in Neptune), writes updated embeddings to ElastiCache. This is what makes real-time Pass 1 feasible — the expensive work is done offline, real-time scoring retrieves the result instantly. Triggered by EventBridge cron.
Batch EmbeddingsMemory
AWS Lambda — Scoring Orchestrator
API Gateway backend · two-pass routing
Why needed: Thin orchestration between API Gateway and the ML endpoints: (1) retrieve embeddings from Redis (~2ms), (2) retrieve profile features from Feature Store (~8ms), (3) invoke TGN Fast Path (~20ms), (4) apply TFLAG two-hurdle deviation filter, (5) if score ≥0.3, invoke SEAL endpoint (+50ms), (6) apply LightGBM meta-learner fusion, (7) return risk tier + explanation signals. Decouples routing logic from ML models — scoring logic updates without redeploying models.
OrchestrationAPI
Layer 5 — Training & Model Management
Amazon SageMaker Training Jobs
ml.g4dn.12xlarge · weekly fine-tune + monthly full retrain
Why needed: Two cadences: Weekly fine-tune (~8h) updates TGN weights on recent labelled transactions (confirmed chargebacks), preserving the pre-trained backbone via LoRA — keeping drift-aligned without catastrophic forgetting. Monthly full retrain (~48h) rebuilds the full ensemble: TGN + RF head + LightGBM meta-learner + drift autoencoder. GraphSMOTE with k-hop topology preservation is applied during training to address <2% fraud rate class imbalance without disrupting temporal ordering.
Training
Amazon SageMaker Pipelines
MLOps orchestration · weekly trigger
Why needed: Orchestrates the full MLOps workflow: data prep → graph construction → TGN training → RF head fitting → meta-learner training → validation → endpoint update. The pipeline enforces a quality gate: AUPRC must be ≥ the current production version before an endpoint update is allowed. Triggered weekly by EventBridge. Provides full lineage tracking, reproducible runs, and integration with Model Registry.
Training
Amazon SageMaker Model Registry
Versioned model store · approval gate
Why needed: Every trained model registered with: training metrics (AUPRC, F1, FP rate per fraud pattern), training data hash for reproducibility, and an approval gate requiring human review before production promotion. Prevents silent model degradation — the comparison is always against the live endpoint, not just an absolute threshold. Also stores per-segment evaluation results (AUPRC separately for card-testing, AML, bust-out) to catch pattern-specific regressions.
Training
AWS Glue ETL
Tabular → graph transformation · nightly
Why needed: Transforms raw transaction records (tabular, in S3) into the heterogeneous graph format Neptune requires (CSV with node/edge files). Runs nightly. Critically, also builds training datasets with strict chronological 70/15/15 splits — no random shuffle. Random splits cause up to 39.5 F1-point inflation on the Elliptic benchmark (confirmed in research). Glue enforces temporal ordering as a pipeline constraint, not an afterthought.

Layer 6 — Drift Detection & Adaptive Maintenance
CloudWatch + SageMaker Model Monitor
Embedding drift · metric monitoring · latency
Why needed: Two-track monitoring: (1) Metric monitoring — rolling AUPRC on labelled feedback (chargebacks 30–90 days post-transaction), endpoint P99 latency, ElastiCache hit rate; (2) Latent-space drift (TGNN-CDD pattern) — autoencoder trained on normal TGN embeddings, reconstruction error monitored via CUSUM. Latent-space drift catches structural and relational drift (new fraud topology, new entity relationships) that raw feature monitoring misses, because it monitors the model's internal representation of normality.
Drift
Amazon EventBridge
Drift triggers · scheduled jobs · pipeline orchestration
Why needed: Event-driven automation for three triggers: (1) CloudWatch alarm (AUPRC drop >5%) → SageMaker Pipeline (retrain); (2) Cron (hourly) → Batch Transform (embedding refresh); (3) Cron (daily) → Glue ETL (feature store update). EventBridge decouples monitoring from action — the retrain trigger fires only when needed (not on a fixed weekly schedule regardless of drift), reducing unnecessary compute.

Layer 7 — API & Integration
Amazon API Gateway (HTTP API)
Core banking interface · IAM auth · throttling
Why needed: Exposes two routes to core banking: POST /score (sync, <100ms P99 via Lambda → Redis + SageMaker) and POST /score/batch (async, via Batch Transform for post-transaction AML review). HTTP API variant is chosen over REST API for lower latency (~1ms vs ~6ms per request overhead). Provides IAM/Cognito auth, per-route throttling to protect SageMaker endpoints, and request logging. Canary deployments for model version rollouts.
API
Amazon S3
Raw data · model artifacts · graph snapshots
Why needed: Central durable store for raw transaction history, training datasets (chronologically split by Glue), graph snapshots (Neptune export), and model artifacts (SageMaker Model Registry). Three-tier lifecycle: S3 Standard (0–90 days, hot), S3-IA (90–365 days, warm), Glacier (365+ days, cold). The training pipeline reads exclusively from S3 to ensure full reproducibility — no live database queries during training.

Layer 8 — Fraud Analyst Tooling
Amazon QuickSight (or CloudFront + AppSync)
Fraud analyst dashboard · fraud ring visualisation
Why needed: Business dashboard for fraud analysts: real-time flagged transaction feed with risk tier and explanation signals, historical fraud pattern trends by type/segment/time-of-day, graph visualisation of detected fraud rings (from Neptune Analytics community detection), and "find similar" via TGN embedding vector similarity search. QuickSight for standard reporting; custom React app on CloudFront + AppSync for interactive graph views with Neptune Analytics.
Analyst
Amazon DocumentDB
Fraud case records · feedback loop
Why needed: Stores enriched fraud case records — flagged transactions with analyst notes, resolution status, chargeback outcomes. Closes the feedback loop: confirmed fraud from DocumentDB becomes a training label in the weekly fine-tune pipeline. This is the only path for supervised labels to re-enter the model — without it, the TGN would drift toward unsupervised operation as fraud patterns evolve. DocumentDB's flexible schema accommodates the heterogeneous case metadata per fraud type.

Key Design Decisions
Neptune + ElastiCache (not just Neptune)
Neptune is the authoritative graph store but P99 Gremlin latency is 50–200ms at inference scale — too slow for the fast path. ElastiCache gives sub-2ms for the 97% of transactions that don't need live graph traversal. The Lambda architecture pattern (BRIGHT, eBay, CIKM 2022) separates batch pre-computation from real-time lookup: Neptune does the rich multi-hop work offline, Redis serves the result instantly.
Two SageMaker endpoints (not one)
Collocating TGN-SEAL with the fast-path TGN would waste GPU capacity on the 97% of transactions that never need structural enrichment. Separate endpoints allow independent auto-scaling: fast-path handles full TPS, SEAL runs on smaller instances since it sees only ~3% of volume. The two-pass architecture (Pass 1: always; Pass 2: flagged only) is what makes sub-100ms P99 achievable at 10K TPS.
SageMaker Feature Store (not just Redis)
Redis stores dynamic embeddings updated per transaction. Feature Store stores slower-changing profile features (account age, balance quantile — inputs to ProfileStateEncoder) and the rolling statistics for the TFLAG two-hurdle filter. Keeping them separate maintains Redis leanness and provides offline/online feature parity for training — a common source of silent AUC degradation in production GNN systems when training and serving use different feature pipelines.
Chronological data splits enforced by Glue (not random)
Research on the Elliptic Bitcoin dataset shows random train/test splits inflate GNN performance by up to 39.5 F1 points vs. a leakage-free chronological split. AWS Glue enforces 70/15/15 chronological ordering as a pipeline constraint. The SageMaker Pipeline quality gate compares AUPRC on the chronological holdout — never on a random split — preventing silent model overestimation from entering production.
