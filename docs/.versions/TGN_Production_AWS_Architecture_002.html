Compute Layer
Amazon API Gateway
Ingress — auth, throttling, routing
Why needed: Single entry point for all scoring requests. Provides request throttling (10k RPS cap to protect SageMaker), JWT authorisation via Cognito (so only authenticated Streamlit sessions and downstream systems can call POST /score), and request/response caching for repeated identical transactions. A WebSocket stage supports real-time score streaming back to the Streamlit UI. Usage Plans give separate rate limits to the Rule Builder UI vs. batch scoring callers.
All modules
AWS Lambda — 3 functions
score-transaction · rule-evaluator · rule-crud
Why needed: Three independent, stateless functions map exactly to the three code boundaries in the codebase: HybridScorer, RuleEvaluator, and the Rule Builder CRUD API. They are kept separate so rule evaluation (~1ms, pure Python) is never blocked waiting for TGN inference (~40ms). Key design rule: rule-evaluator fires first; a BLOCK verdict short-circuits the flow and SageMaker is never invoked — saving cost and latency. Each function has its own IAM execution role scoped to only the resources it touches.

Amazon SageMaker Real-Time Endpoint
TGN PyTorch model — GPU inference
Why needed: Lambda cannot run the TGNFraudDetector — it's a PyTorch GNN with a 512MB+ model state and requires a GPU for sub-100ms latency. SageMaker hosts the trained .pt checkpoint from S3 on a ml.g4dn.xlarge instance. The endpoint exposes a JSON API that the score-transaction Lambda calls via sagemaker-runtime:InvokeEndpoint. Application Auto Scaling scales the endpoint out on invocation count, and multi-model endpoints allow both the Standard TGN and Ensemble TGN checkpoints to co-reside on the same instance to reduce cost.

AWS Batch + ECR
Nightly embedding pre-computation
Why needed: The BatchEmbedder in ensemble/embedding/batch_embedder.py runs the full temporal graph to pre-compute entity embeddings. This job takes 10–30 minutes depending on graph size — well beyond Lambda's 15-minute limit — and needs GPU. The Dockerised PyTorch job is stored in ECR and run as an AWS Batch job on a spot-priced ml.g4dn.xlarge compute environment triggered nightly by EventBridge Scheduler. Outputs are written to S3 and synced to ElastiCache by a post-job Lambda.

Caching Layer
Amazon ElastiCache (Redis)
Sub-ms entity embedding lookup
Why needed: The TGN Lambda Architecture (from ENSEMBLE_DESIGN.md) separates offline embedding pre-computation from real-time inference. At scoring time, the score-transaction Lambda fetches emb:{src_id} and emb:{dst_id} from Redis (each a float32 vector) before calling SageMaker. This avoids full neighbourhood expansion at inference time, cutting P99 latency from ~200ms to ~50ms. Keys have a 24-hour TTL refreshed by the nightly Batch job. Redis Cluster mode with 3 shards provides horizontal scale.
Embedding Cache TGN Scorer
Amazon DAX (DynamoDB Accelerator)
Microsecond rule set reads
Why needed: The rule-evaluator Lambda reads the full active rule set on every transaction. Without caching, this would hammer DynamoDB at 10k+ TPS. DAX sits in front of the fraud-rules DynamoDB table and serves the rule set from in-memory cache with microsecond reads. The rule set is cached with a 5-minute TTL — acceptable staleness for rules that rarely change. When rule-crud Lambda updates a rule, it writes through DAX, ensuring cache invalidation is immediate.

Storage Layer
Amazon DynamoDB — 2 tables
fraud-rules · score-audit
Why needed: Two tables serve distinct purposes on the hot path. fraud-rules stores rule definitions (conditions JSON, verdict, priority, enabled flag) — the Rule Builder UI reads and writes here via rule-crud Lambda. score-audit records every scoring decision (tx_id, TGN score, rule verdict, final tier, timestamp) for compliance, model monitoring, and drift detection. DynamoDB Streams on fraud-rules trigger DAX cache invalidation; Streams on score-audit feed the CloudWatch drift metrics pipeline.

Amazon S3 — 3 buckets
model-artifacts · transaction-data · embeddings-batch
Why needed: Three buckets map to three data lifecycle stages. tgn-model-artifacts stores versioned .pt checkpoints produced by SageMaker Pipelines — SageMaker loads the model from here on endpoint deployment. tgn-transaction-data receives raw CSV uploads from the Upload CSV page (page 5), partitioned by date; an S3 Event Notification triggers a retraining pipeline check when new labelled data arrives. tgn-embeddings-batch is the staging area between the nightly Batch job and ElastiCache sync. All buckets use S3 Intelligent-Tiering and Server-Side Encryption (SSE-KMS).

AWS Secrets Manager
Credential management for all Lambdas
Why needed: Lambda functions need Redis connection strings, SageMaker endpoint ARNs, and DynamoDB table names without hardcoding. Secrets Manager stores these as versioned secrets and rotates them automatically. Lambdas fetch at cold-start and cache in-memory for the function lifetime, so there's no per-invocation secrets call. Resource-based policies restrict each secret to only the Lambda IAM role that needs it — e.g., only score-transaction can read the Redis URL, not rule-evaluator.

ML Platform & MLOps
Amazon SageMaker Pipelines + Model Registry
Automated retraining & deployment
Why needed: When new labelled fraud data arrives in S3, the TGNTrainer workflow must rerun automatically. SageMaker Pipelines defines the steps: Data Validation → Training (running tgn_learn/training/trainer.py) → Evaluation (AUPRC ≥ threshold check) → Conditional Registration (only Approved models proceed) → Endpoint Update. Model Registry maintains full version history so you can roll back to a previous checkpoint instantly. Triggered by EventBridge when labelled data volume exceeds a configurable threshold.
MLOps Pipeline TGN Scorer
SageMaker Model Monitor
Data drift + prediction quality monitoring
Why needed: The drift_monitor.py and threshold_adapter.py components in ensemble/maintenance/ are the production version of the manual drift detection in the Streamlit app. Model Monitor runs as a scheduled job capturing the endpoint's inference inputs/outputs, comparing them against a baseline captured at training time. When feature distribution shift (e.g. amount ranges changing due to inflation) or prediction drift (model confidence degrading) is detected, CloudWatch alarms fire and the retraining pipeline is triggered automatically.

Orchestration & Scheduling
Amazon EventBridge Scheduler
Nightly batch + conditional retraining triggers
Why needed: EventBridge is the decoupling layer between all async workflows. Three rules: (1) cron(0 2 * * ? *) triggers the nightly AWS Batch embedding job; (2) S3 PutObject events on tgn-transaction-data trigger a retraining eligibility check in SageMaker Pipelines; (3) DynamoDB Streams from fraud-rules trigger a Lambda that invalidates DAX cache when a rule is updated. EventBridge's dead-letter queue support ensures missed triggers are retried.

AWS Step Functions (Express)
Batch scoring orchestration
Why needed: The Batch Score tab (page 4, tab_batch) scores thousands of transactions from a CSV upload. In production, this is too large for a single Lambda invocation. A Step Functions Express Workflow fans out: S3 CSV → SQS chunks → parallel Lambda invocations → aggregated results written back to S3. Express Workflows are priced per-execution (not per-state-transition), making them cost-effective for high-throughput batch jobs. Results are streamed back to the UI via S3 presigned URLs.

Security & Access Control
Amazon Cognito
Role-based access to Rule Builder UI
Why needed: The Rule Builder (page 8) must enforce two access tiers: Analyst (create/edit/test rules, cannot enable BLOCK verdict) and Risk Manager (full access including enabling hard-block rules). Cognito User Pools provide the user directory and JWT token issuing; API Gateway validates the token on every rule-crud call and checks the role claim before allowing a BLOCK verdict rule to be enabled. This prevents an analyst from accidentally (or maliciously) blocking all transactions with a misconfigured rule.
Rule Builder UI API Gateway
AWS KMS (CMK)
Encryption at rest — transactions & model artefacts
Why needed: Transaction data and fraud labels are regulated financial data. All three S3 buckets and both DynamoDB tables must use Customer Managed Keys (CMK) — not AWS-managed keys — so that every decryption event appears in CloudTrail with a full audit trail. Separate CMKs per data tier (raw transactions vs. model artefacts vs. audit log) means a compromised key for one tier doesn't expose the others. Key policies restrict usage to only the Lambda and SageMaker IAM roles that legitimately need access.
S3 · DynamoDB · ElastiCache
Amazon VPC + Interface Endpoints
Private network — no public internet for data plane
Why needed: All Lambda functions, the SageMaker endpoint, ElastiCache, and DynamoDB should reside in a private VPC with no internet gateway. AWS service traffic (S3, DynamoDB, SageMaker, Bedrock, Secrets Manager) routes through VPC Interface Endpoints — never traversing the public internet. This eliminates the attack surface of data exfiltration via the network path. SageMaker notebooks for model development live in a separate VPC peered to the production VPC with appropriate Security Group rules.
All services
Amazon GuardDuty + AWS WAF
Threat detection on the system itself
Why needed: A fraud detection system is itself a target. GuardDuty monitors for anomalous API calls (e.g., unexpected sagemaker:InvokeEndpoint from an unrecognised principal, S3 exfiltration patterns on the model artefacts bucket). AWS WAF sits in front of API Gateway to block SQL injection, malformed JSON payloads, and rate-abuse from IP ranges. GuardDuty findings feed into Security Hub alongside Inspector (container image vulnerability scanning for the Batch ECR image) for a unified security posture view.

Observability
Amazon CloudWatch (Logs + Metrics + Alarms)
Model drift + latency + rule health
Why needed: Four custom metric dimensions map to the drift signals in drift_monitor.py and threshold_adapter.py: FraudRate_7d rolling average (population shift detection), TGNScore_p95 (model confidence degradation), RuleHitRate per rule ID (stale or over-triggering rules), and SageMakerEndpoint/Latency (inference slowdown). Embedded Metric Format (EMF) in Lambda handlers writes structured log lines that become queryable metrics automatically — zero overhead. CloudWatch Dashboards give the SOC a live view of fraud rate, block rate, and model health.

AWS X-Ray
Distributed tracing across Lambda & SageMaker
Why needed: The scoring path chains API Gateway → two Lambda invocations → ElastiCache → SageMaker. X-Ray traces the complete path of a single transaction through all hops — critical for diagnosing P99 latency spikes (e.g., "Redis cold key added 15ms on this call" or "SageMaker scaled down and the cold start added 300ms"). Enable Active Tracing on both Lambda functions and the API Gateway stage. The trace shows exactly where time is spent, which is especially important when debugging the BLOCK path (should be <2ms end-to-end).

Amazon SNS
Drift alerts & MLOps notifications
Why needed: CloudWatch Alarms publish to SNS topics which fan out to multiple subscribers: email for the risk team (fraud rate spike), Slack webhook for the ML team (model drift detected, retraining triggered), and a PagerDuty Lambda for P0 incidents (endpoint unavailable, all transactions returning CRITICAL). SNS decouples alarm firing from notification routing — you can add new subscribers (SIEM, ticketing system) without changing the alarms. Separate topics per severity (INFO, WARNING, CRITICAL) prevent alert fatigue.

AWS CDK (Python) — 5 stacks

Why needed: All infrastructure deployed as code from a cdk/ directory in the repo root, matching the isolation architecture principle in ENSEMBLE_DESIGN.md. 2

4
HybridScorer returns synthetic CRITICAL result immediately
5
